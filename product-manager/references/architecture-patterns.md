# Vibe Coding 时代的架构决策与技术选型指南

> 产品经理不需要能写 Terraform 配置，但需要理解架构决策背后的 trade-off。

> 本文档帮你在 vibe coding 时做出不会后悔的架构决策。

## 1. 架构决策原则

### Vibe Coding 时代的三条基本原则

**原则一：托管优先于自建**

AI 生成的代码不会帮你运维。选择托管服务（Supabase、Vercel、Railway）意味着你不需要关心服务器维护、安全补丁、扩容这些事。成本略高，但省下的时间和避免的风险远超成本。

**原则二：约束先于自由**

AI 在有明确约束的环境中表现最好。选技术栈时，选那些"有明确最佳实践"的，而不是"最灵活"的。例如 Next.js 的文件路由 > Express 的自由路由——因为 AI 在约定俗成的模式中犯错更少。

**原则三：可替换优于深耦合**

AI 代码的特点是"能跑但可能需要重写"。架构设计要让各模块可以独立替换——今天 AI 写的认证模块不好用？换掉它不应该影响其他模块。

## 2. 数据库选型决策树

```
你的数据需要什么？
│
├─ 结构化数据（用户、订单、商品）
│  ├─ 需要关系查询（JOIN）？
│  │  ├─ 是 → PostgreSQL（via Supabase）
│  │  │       ✅ 内置 Auth、RLS、实时订阅
│  │  │       ✅ 最成熟的开源关系数据库
│  │  │       ✅ AI 生成 SQL 的质量最高（训练数据最多）
│  │  │       ⚠️  复杂查询的性能需要关注索引
│  │  │
│  │  └─ 不确定 → 也选 PostgreSQL（最安全的默认选择）
│  │
│  └─ 不需要关系，纯 key-value？
│     └─ Redis（缓存层）或 DynamoDB（持久化）
│
├─ 非结构化/半结构化数据（日志、配置、灵活 schema）
│  └─ MongoDB
│      ⚠️  小心：AI 生成的 MongoDB 代码更容易有 NoSQL 注入漏洞
│      ⚠️  没有严格 schema 意味着数据质量完全依赖应用层
│
├─ 实时协作数据（聊天、协作文档、在线状态）
│  └─ Firebase Realtime DB 或 Supabase Realtime
│
├─ 向量/语义搜索（AI 相关功能）
│  └─ PostgreSQL + pgvector 或 Pinecone
│
└─ 文件/媒体存储
   └─ 永远不要存数据库！
      → S3 / Supabase Storage / Cloudflare R2
      → 数据库只存文件的 URL/路径
```

### 数据库配置安全清单

无论选什么数据库，这些配置必须做：

| 配置项 | 说明 | 优先级 |
|--------|------|--------|
| 连接加密（SSL） | 数据传输不被窃听 | 必须 |
| 最小权限账号 | 应用使用的数据库账号只有必要的权限，不用 root | 必须 |
| 行级安全（RLS） | 用户 A 看不到用户 B 的数据 | 必须（如果有多用户） |
| 连接池 | 限制同时连接数，防止连接耗尽 | 建议 |
| 自动备份 | 至少每天一次，保留 7 天 | 必须 |
| 慢查询日志 | 识别性能瓶颈 | 建议 |

## 3. 前端架构选型

### 框架选择

```
你在做什么类型的产品？
│
├─ 内容型网站（博客、文档、营销页）
│  └─ Next.js（SSG/SSR）或 Astro
│      ✅ SEO 友好
│      ✅ 首屏加载快
│
├─ 交互型应用（SaaS、Dashboard、管理后台）
│  └─ Next.js（App Router）或 Remix
│      ✅ 服务端渲染 + 客户端交互
│      ✅ 文件路由约定，AI 代码结构更规范
│
├─ 移动端优先
│  └─ React Native（如果需要原生功能）
│  └─ 响应式 Web（如果 PWA 就够用）
│
└─ 快速原型 / 一次性 demo
   └─ 纯 HTML + Tailwind + Alpine.js
       ✅ 零构建，最快上手
       ⚠️  不适合长期维护的产品
```

### 前端安全要点

这些是 AI 最容易忽略的前端安全问题：

| 问题 | AI 经常做的 | 你应该要求的 |
|------|-----------|------------|
| 状态管理 | 把用户 token 存 localStorage | token 存 HTTP-only cookie，前端不直接接触 |
| 权限控制 | 前端 `if (role === 'admin')` 控制显示 | 前端只控制 UI 展示，真正的权限检查在 API 层 |
| 敏感数据 | 把完整的用户对象存到前端 state | API 只返回前端需要的字段，不多也不少 |
| 环境变量 | 把所有环境变量都暴露给前端 | 前端只能访问 `NEXT_PUBLIC_` 前缀的变量 |
| 第三方脚本 | 随意引入 CDN 上的 JS | 使用 CSP 限制脚本来源，只引入可信来源 |

## 4. 后端架构选型

### API 架构模式

| 模式 | 适用场景 | PM 需要了解的 |
|------|---------|-------------|
| **REST API** | 大多数应用的默认选择 | 最成熟、AI 生成质量最高、工具最丰富 |
| **GraphQL** | 前端需要灵活查询不同数据组合 | 更复杂，AI 容易生成有安全问题的 resolver |
| **tRPC** | TypeScript 全栈应用 | 类型安全，但生态小，AI 经验少 |
| **Serverless Functions** | 简单的 API、webhook 处理 | 适合 vibe coding，但注意冷启动和超时 |

对于 vibe coding 项目的建议：**默认选 REST + Next.js API Routes**。理由是 AI 对 REST 模式的训练数据最充分，犯错率最低。

### 认证架构

```
认证方案选择：
│
├─ 最简单（适合 MVP）
│  └─ NextAuth.js / Auth.js
│      ✅ 与 Next.js 深度集成
│      ✅ 支持 OAuth（Google、GitHub 等）
│      ✅ 内置 session 管理
│      ⚠️  自定义困难
│
├─ 中等复杂度（适合正式产品）
│  └─ Supabase Auth 或 Clerk
│      ✅ 托管服务，不需要自己维护
│      ✅ 预置 UI 组件（登录页、注册页）
│      ✅ 包含 MFA、角色管理
│
├─ 最灵活（需要工程师参与）
│  └─ 自建 JWT + bcrypt + Redis session
│      ✅ 完全可控
│      ⚠️  自己写认证 = 自己扛安全风险
│      ⚠️  AI 生成的认证代码漏洞率最高
│
└─ 绝对不要
   └─ 前端存密码、明文 token、自制加密
```

### 后端安全架构

```
请求处理的安全洋葱模型（由外到内每层都要有防护）：

[Layer 1: 网络层]  → CDN / WAF / DDoS 防护
    ↓
[Layer 2: 入口层]  → Rate limiting / IP 过滤 / CORS
    ↓
[Layer 3: 认证层]  → JWT 验证 / Session 校验
    ↓
[Layer 4: 授权层]  → 角色检查 / 资源所有权验证
    ↓
[Layer 5: 校验层]  → 输入校验（zod/joi）/ 参数清洗
    ↓
[Layer 6: 业务层]  → 业务逻辑
    ↓
[Layer 7: 数据层]  → 参数化查询 / RLS / 最小权限
    ↓
[Layer 8: 输出层]  → 响应过滤（只返回需要的字段）/ 错误处理
```

AI 生成的代码通常只实现 Layer 6（业务层），其他层需要你在 PRD 中明确要求。

## 5. 部署架构选型

### 部署平台对比

| 平台 | 适合场景 | 免费额度 | PM 友好度 | 备注 |
|------|---------|---------|-----------|------|
| **Vercel** | Next.js 应用 | 充足（hobby plan） | ⭐⭐⭐⭐⭐ | git push 即部署 |
| **Railway** | 需要后端服务/数据库 | $5/月 credit | ⭐⭐⭐⭐ | 支持任意语言 |
| **Supabase** | 需要 BaaS（数据库+认证+存储） | 2 个免费项目 | ⭐⭐⭐⭐⭐ | 开源 Firebase 替代 |
| **Cloudflare Pages** | 静态站点 + Edge Functions | 非常大方 | ⭐⭐⭐⭐ | 全球 CDN |
| **Fly.io** | 需要 Docker 容器 | 部分免费 | ⭐⭐⭐ | 更灵活但更复杂 |

### 环境隔离（这条是铁律）

```
永远保持三套环境完全隔离：

[开发环境 (dev)]     → 本地开发用，数据是假的
     ↓ 代码合并
[预发布环境 (staging)] → 上线前验证，数据接近真实但是脱敏的
     ↓ 通过验证
[生产环境 (production)] → 用户使用的真实环境

铁律：
✅ 每套环境有独立的数据库
✅ 每套环境有独立的环境变量
✅ 每套环境有独立的第三方服务配置（如 Stripe test key vs live key）
❌ 永远不要让开发环境连接生产数据库
❌ 永远不要在生产环境运行测试
❌ 永远不要用生产数据做开发调试（如必须，先脱敏）
```

## 6. 常见架构反模式

AI 在 vibe coding 中经常产生的架构级别错误：

### 反模式 1：一切皆前端

**表现**：所有逻辑都在浏览器中运行——包括数据库查询、API 密钥、业务规则。

**为什么 AI 这么做**：因为这是最"直接"的实现方式，一个文件搞定一切。

**后果**：用户可以看到你的所有代码，包括密钥、业务逻辑、数据库连接。

**修复**：任何涉及密钥、权限、数据库的逻辑都必须在服务端。

### 反模式 2：巨石式单文件

**表现**：AI 把所有代码写在一个巨大的文件里——3000 行的 `app.js`。

**为什么 AI 这么做**：单文件意味着不需要处理模块间的导入导出关系。

**后果**：无法维护，无法测试，下次对话 AI 也无法理解这个文件。

**修复**：在 PRD 中要求模块化，明确目录结构。

### 反模式 3：框架混搭

**表现**：项目中同时使用 Express + Koa + Hono，或 React + Vue 组件混用。

**为什么 AI 这么做**：不同对话中 AI 可能选择不同的框架，缺乏全局一致性。

**后果**：依赖冲突、运行时错误、维护噩梦。

**修复**：在项目约束文件（AGENTS.md / CLAUDE.md）中锁定技术栈。

### 反模式 4："临时"变"永久"

**表现**：`// TODO: fix later`、`// HACK: this works for now`、`password = "admin123"`。

**为什么 AI 这么做**：AI 的首要任务是让代码"能运行"。临时方案最快。

**后果**：在 vibe coding 的速度下，"later" 永远不会到来。代码带着临时方案上线。

**修复**：在 PRD 中明确不允许临时方案。AI 要么正确实现，要么标记为"需要讨论"并暂停。

### 反模式 5：数据库里塞一切

**表现**：把用户上传的图片、PDF 文件直接以 blob 存到数据库中。

**为什么 AI 这么做**：这是让功能"跑起来"最快的方式，不需要配置额外的存储服务。

**后果**：数据库急剧膨胀、查询变慢、备份困难、成本飙升。

**修复**：文件存对象存储（S3/Supabase Storage），数据库只存 URL。

## 7. 从原型到生产的迁移路径

vibe coding 做原型非常快。但如果验证了产品想法，准备认真做产品时，需要一个系统化的迁移过程：

### 迁移清单

```
Phase 0: 评估原型（1-2 天）
  - [ ] 列出原型中所有的"临时方案"和 TODO
  - [ ] 评估哪些代码可以复用、哪些必须重写
  - [ ] 确认数据模型是否需要调整

Phase 1: 基础设施（2-3 天）
  - [ ] 建立三套环境（dev / staging / prod）
  - [ ] 配置 CI/CD 流水线
  - [ ] 配置环境变量管理
  - [ ] 设置数据库备份

Phase 2: 安全加固（2-3 天）
  - [ ] 运行安全检查清单（references/security-checklist.md）
  - [ ] 移除所有硬编码密钥
  - [ ] 实现完整的认证授权
  - [ ] 添加 rate limiting 和输入校验

Phase 3: 代码质量（3-5 天）
  - [ ] 建立代码规范（ESLint + Prettier）
  - [ ] 添加 TypeScript 严格模式
  - [ ] 编写核心功能的测试
  - [ ] 重构不一致的代码模式

Phase 4: 监控与运维（1-2 天）
  - [ ] 配置错误监控（Sentry）
  - [ ] 配置可用性监控（UptimeRobot）
  - [ ] 配置日志系统
  - [ ] 建立告警机制

Phase 5: 文档（1 天）
  - [ ] 更新 PRD（反映最终实现，而不是最初设想）
  - [ ] 写部署文档
  - [ ] 写 API 文档（可以用 AI 生成 OpenAPI spec）
  - [ ] 写 onboarding 文档（新团队成员如何上手）
```

### 成本参考

用 vibe coding 做一个 MVP 到生产级产品的典型成本（独立产品经理 / 小团队）：

| 阶段 | 时间 | 月成本 |
|------|------|--------|
| 原型（vibe coding） | 1-3 天 | $0（免费额度内） |
| 安全加固 + 质量提升 | 1-2 周 | $0-50（工具费用） |
| 生产环境运行 | 持续 | $20-100/月（Vercel + Supabase + Sentry） |
| 安全审计（如需要） | 按次 | $500-5000（外部审计） |

注意：如果产品涉及支付、医疗、金融等高合规要求的领域，安全审计不是可选项。
