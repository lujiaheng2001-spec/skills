## 1. 风险评估矩阵

每个功能上线前，用这个矩阵评估风险等级：

### 影响度（如果出问题会怎样）

| 等级 | 定义 | 示例 |
|------|------|------|
| **致命** | 数据泄露、资金损失、法律风险 | 用户密码泄露、支付金额被篡改 |
| **严重** | 核心功能不可用、大量用户受影响 | 登录系统崩溃、数据丢失 |
| **中等** | 部分功能异常、少量用户受影响 | 搜索结果不准确、页面偶尔报错 |
| **轻微** | 体验问题、不影响核心功能 | 样式错乱、文案错误 |

### 可能性（出问题的概率有多大）

| 等级 | 定义 | 信号 |
|------|------|------|
| **极高** | AI 生成后未经任何审查 | 代码直接提交，没有测试，没有 review |
| **高** | 只做了基本的功能测试 | "能跑就行"，没有测过边界情况和异常 |
| **中** | 做了系统性测试但缺少安全审查 | 功能正常，但没有检查认证、权限、注入 |
| **低** | 经过完整的测试和安全检查 | 跑了自动化测试 + 安全扫描 + 人工 review |

### 决策矩阵

```
                    影响度
                致命    严重    中等    轻微
可能性  极高  │ 禁止上线 │ 禁止上线 │ 需修复  │ 可观察  │
        高    │ 禁止上线 │ 需修复   │ 需修复  │ 可观察  │
        中    │ 需修复   │ 需修复   │ 可观察  │ 可接受  │
        低    │ 可观察   │ 可接受   │ 可接受  │ 可接受  │
```

- **禁止上线**：必须修复后才能部署
- **需修复**：本次迭代内必须修复
- **可观察**：上线后密切监控，下次迭代修复
- **可接受**：记录在案，排入 backlog

## 2. 按功能类型的风险图谱

不同类型的功能，风险分布完全不同。以下是 vibe coding 最容易出问题的功能类型排名：

### 🔴 高风险（建议有工程师参与）

| 功能类型 | 核心风险 | 常见 AI 生成的坑 |
|---------|---------|----------------|
| **支付/金融** | 金额篡改、重复扣款、退款漏洞 | 金额计算用浮点数而不是整数（分）、没有幂等性检查 |
| **认证/权限** | 未授权访问、账号被盗 | 权限检查写在前端、token 存 localStorage、没有 rate limit |
| **文件上传** | 远程代码执行、服务器被控 | 不校验文件类型、不限制大小、上传路径可遍历 |
| **用户数据管理** | PII 泄露、GDPR 违规 | 过度收集数据、日志打印敏感信息、没有数据删除功能 |
| **API 对外暴露** | 数据泄露、被恶意调用 | 没有认证、没有限流、响应包含过多数据 |

### 🟡 中等风险（充分测试后可以自行处理）

| 功能类型 | 核心风险 | 注意事项 |
|---------|---------|---------|
| **搜索/过滤** | SQL 注入、性能问题 | 使用 ORM 的参数化查询，大数据量要加分页和索引 |
| **表单提交** | XSS、数据校验绕过 | 前后端双重校验，使用成熟的 schema 校验库（zod/joi） |
| **实时功能** | WebSocket 安全、连接泄漏 | WebSocket 也需要认证，断线要有重连和清理机制 |
| **邮件/通知** | 被用作垃圾邮件中继 | 限制发送频率，校验收件人是否合法 |

### 🟢 低风险（PM 可以独立完成）

| 功能类型 | 注意事项 |
|---------|---------|
| **静态内容展示** | 确保 CSP 配置正确，图片做好懒加载 |
| **前端交互/动画** | 注意性能，别在移动端卡顿 |
| **营销落地页** | SEO meta tags、Open Graph 配置 |
| **内部工具/dashboard** | 确保不对外暴露，加基本认证 |

## 3. 预警信号清单

在开发过程中，如果观察到以下信号，立即停下来评估：

### 代码层面

| 预警信号 | 意味着什么 | 行动 |
|---------|----------|------|
| AI 说"为了简化，我先这样写" | 它在走捷径，通常是安全捷径 | 要求它按正确方式实现 |
| 代码中出现 `// TODO: add auth later` | "later"在 vibe coding 里意味着"永远不会" | 现在就加 |
| `npm install` 安装了 20+ 个你没见过的包 | 可能有幻觉包或过度依赖 | 逐个确认必要性 |
| 代码在不同文件中用不同方式处理同一件事 | AI 没有全局一致性，每次对话重新发明轮子 | 抽取公共模块 |
| 测试全部 pass 但只有个位数测试 | 测试覆盖不足，假的安全感 | 补充边界case和异常测试 |

### 架构层面

| 预警信号 | 意味着什么 | 行动 |
|---------|----------|------|
| 环境变量在代码中有默认值 | 意味着即使忘记配置环境变量，程序也会用不安全的默认值运行 | 缺少环境变量时程序应该拒绝启动 |
| 测试库和生产库用同一个数据库 | 测试数据可能污染生产，或测试时意外删除生产数据 | 立即分离 |
| 所有数据都存在一个表里 | AI 图省事儿，没有正确建模实体关系 | 重新设计 schema |
| API 没有版本号 | 后续改接口会 break 所有客户端 | 加 `/api/v1/` 前缀 |
| 没有错误处理中间件 | 任何未捕获的错误都会导致 500 + stack trace 暴露 | 添加全局错误处理 |

### 流程层面

| 预警信号 | 意味着什么 | 行动 |
|---------|----------|------|
| "先上线再优化" | 可能合理，但如果涉及安全和数据，这是错误的 | 安全和数据问题必须在上线前解决 |
| 跳过 code review 直接合并 | 省了 30 分钟，可能换来 300 小时的修复 | 至少做一次快速 review |
| "这个功能很简单，不需要测试" | 最危险的假设。"简单"功能经常有意想不到的边界情况 | 至少写一个 happy path 测试 |
| 上线后没有监控 | 出了问题你是最后一个知道的人 | 配置 Sentry + UptimeRobot |

## 4. 技术债评估模型

### 技术债的"利率"

不是所有技术债都相等。一些债务的"利率"很高——越不处理，后果越严重：

| 债务类型 | 利率 | 说明 |
|---------|------|------|
| 安全漏洞 | 极高 | 每多一天就多一天被攻击的风险 |
| 无测试的核心逻辑 | 高 | 每次改动都可能引入 bug 且无法发现 |
| 不一致的代码风格 | 中 | 影响开发效率，但不会直接导致故障 |
| 缺少文档 | 中 | 换人或回来维护时会很痛苦 |
| UI 瑕疵 | 低 | 影响体验但不影响功能 |
| 性能未优化 | 视情况 | 用户量小时无所谓，用户量大时是炸弹 |

### 债务清理优先级

```
紧急修复（本周内）：
  → 安全漏洞、数据泄露风险、认证绕过

短期清理（本月内）：
  → 缺失的核心测试、硬编码的配置、环境混用

中期偿还（本季度）：
  → 代码重构、依赖更新、文档补全

长期规划（随产品迭代）：
  → 架构优化、性能提升、技术栈升级
```

## 5. 什么时候必须找专业工程师

PM 用 vibe coding 做很多事都没问题，但某些场景需要专业人员参与。这不是能力问题——这是风险管理。

### 必须找工程师的场景

1. **涉及支付**：金额计算、支付流程、退款逻辑——一个小数点的错误就是真金白银
2. **涉及加密**：自己实现加密算法是安全领域最大的反模式。使用经过验证的库，最好由懂密码学的人配置
3. **涉及合规**：GDPR、HIPAA、PCI DSS——这些法规有具体的技术要求，违规的罚款是收入的百分比
4. **高并发场景**：当你预期有大量同时在线用户时，架构设计需要专业考量
5. **涉及用户安全**：医疗应用、自动驾驶、金融交易——任何错误可能造成人身或财产损害
6. **数据迁移**：从旧系统迁移到新系统，数据丢失可能是不可逆的

### 可以自行处理的场景

1. 内部工具和 dashboard
2. 营销页面和落地页
3. MVP 原型验证（明确是原型，不是最终产品）
4. 内容管理后台
5. 数据分析脚本
6. 自动化工作流

## 6. 上线风险评估模板

每次上线前，填写这个模板，留档备查：

```markdown
# [功能名称] 上线风险评估

## 基本信息
- 评估人：
- 评估日期：
- 计划上线日期：
- 开发方式：[Vibe coding / 人工开发 / 混合]

## 功能概述
[一段话说明这个功能做什么]

## 风险评估

### 安全风险
- [ ] 是否涉及用户认证/权限？ → 如是，风险 +2
- [ ] 是否涉及支付/金融？ → 如是，风险 +3
- [ ] 是否处理用户 PII？ → 如是，风险 +2
- [ ] 是否有文件上传功能？ → 如是，风险 +1
- [ ] 是否暴露新的 API？ → 如是，风险 +1

总分：___  / 风险等级：[低(0-2) / 中(3-5) / 高(6-9)]

### 数据风险
- [ ] 是否修改了数据库 schema？ → 如是，migration 是否可回滚？
- [ ] 是否改变了数据的访问模式？ → 如是，RLS 是否更新？
- [ ] 是否涉及数据删除？ → 如是，是否有备份和恢复方案？

### 回滚方案
- 回滚方式：[重新部署上一版本 / Feature flag 关闭 / 数据库 rollback]
- 预计回滚时间：[X 分钟]
- 回滚后的副作用：[描述]

### 检查确认
- [ ] 安全检查清单已完成（references/security-checklist.md）
- [ ] 核心流程 E2E 测试通过
- [ ] 跨账号权限测试通过
- [ ] 监控和报警已配置
- [ ] 回滚方案已验证

## 结论
- [ ] 可以上线
- [ ] 需修复后上线（附问题列表）
- [ ] 不建议上线（附原因）
```
